Script Management Commands

Script Management Commands

Overview

Quill provides a comprehensive set of commands for managing scripts in your Minecraft server. Scripts can be loaded, reloaded, unloaded, edited, and listed. Scripts are organized into global scripts (available server-wide) and scope scripts (restricted to specific regions and owners).

/quill load

Load a Quill script from the scripts directory and execute it.

Syntax

/quill load <filename>

Parameters

  • <filename> - The name of the script file to load
  • Can be specified with or without the .ql or .quill extension
  • For scope scripts, use the format: <scope_name>/<filename>
  • Examples: myscript, myscript.ql, myScope/events.ql

Permissions

  • Global Scripts: quill.script.load.global (default: op)
  • Scope Scripts: quill.script.load.scope (default: true)
  • Scope owners can always load scripts in their own scopes

Examples

/quill load welcome.ql
/quill load myscript
/quill load myScope/events.ql

/quill reload

Reload a currently loaded script. This unloads the script and loads it again with any changes you've made to the file.

Syntax

/quill reload <filename>

Parameters

  • <filename> - The name of the script file to reload
  • Can be specified with or without the .ql or .quill extension
  • For scope scripts, use the format: <scope_name>/<filename>

Permissions

  • Global Scripts: quill.script.reload.global (default: op)
  • Scope Scripts: quill.script.reload.scope (default: true)
  • Scope owners can always reload scripts in their own scopes

Examples

/quill reload welcome.ql
/quill reload myScope/events

/quill unload

Unload a currently loaded script, removing its event handlers and stopping execution.

Syntax

/quill unload <filename>

Parameters

  • <filename> - The name of the script file to unload
  • Can be specified with or without the .ql or .quill extension
  • For scope scripts, use the format: <scope_name>/<filename>

Permissions

  • Global Scripts: quill.script.unload.global (default: op)
  • Scope Scripts: quill.script.unload.scope (default: true)
  • Scope owners can always unload scripts in their own scopes

Examples

/quill unload welcome
/quill unload myScope/events.ql

/quill list

List all available scripts in the scripts directory, showing their load status.

Syntax

/quill list

Permissions

  • Basic Listing: quill.script.list (default: true)
  • Shows global scripts and scripts in scopes you own
  • List All: quill.script.list.all (default: op)
  • Shows all scripts including those in other players' scopes

Output

The command displays:
- [LOADED] - Script is currently loaded and active (green)
- [UNLOADED] - Script exists but is not loaded (gray)
- Script path (e.g., welcome.ql or myScope/events.ql)

Examples

/quill list

Sample Output:

=== Available Quill Scripts ===
[LOADED] welcome.ql
[UNLOADED] test.ql
[LOADED] playerScope/events.ql
[UNLOADED] adminScope/cleanup.ql
==============================

/quill edit

Open a script in the web-based Quill editor for live editing. Changes are automatically saved back to the server.

Syntax

/quill edit <filename>

Parameters

  • <filename> - The name of the script file to edit
  • Can be specified with or without the .ql or .quill extension
  • For scope scripts, use the format: <scope_name>/<filename>
  • If the file doesn't exist, it will be created

Permissions

  • Global Scripts: quill.script.edit.global (default: op)
  • Scope Scripts: quill.script.edit.scope (default: true)
  • Scope owners can always edit scripts in their own scopes

Web Editor

When you run /quill edit, the command:
1. Creates a temporary editing session
2. Provides a clickable link to the web editor
3. Loads the current script content into the editor
4. Waits for you to save your changes
5. Automatically writes the changes back to the server
6. Notifies you when the save is complete

The editor URL is configured in config.yml:

editor:
  url: https://quill.kmathers.co.uk/editor/#

Examples

/quill edit welcome.ql
/quill edit myScope/events
/quill edit newscript

Script File Organization

Global Scripts

Global scripts are placed directly in the plugins/Quill/scripts/ directory and run server-wide with full permissions (unless restricted by the global scope configuration).

Location:

plugins/Quill/scripts/
├── welcome.ql
├── broadcast.ql
└── admin-tools.ql

Characteristics:
- Available to all players
- Require op permissions to load/reload/unload/edit
- Execute in the "global" scope with unrestricted region boundaries
- Event handlers trigger for all matching events server-wide

Scope Scripts

Scope scripts are organized in subdirectories named after their scopes and are restricted to specific regions and permissions.

Location:

plugins/Quill/scripts/
└── myScope/
    ├── events.ql
    ├── utilities.ql
    └── protections.ql

Characteristics:
- Only execute within the scope's defined region boundaries
- Owned by a specific player (UUID)
- Subject to whitelist/blacklist security mode
- Scope owners can manage their own scripts without op permissions
- Event handlers only trigger for events within the scope's boundaries

File Naming

Valid Extensions:
- .ql - Standard Quill script extension
- .quill - Alternative extension

Naming Convention:
- Use lowercase letters, numbers, underscores, and hyphens
- Avoid spaces in filenames
- Be descriptive: player-join-events.ql is better than pje.ql

Examples:

welcome.ql              ✓ Good
player_events.ql        ✓ Good
admin-tools.ql          ✓ Good
my script.ql            ✗ Bad (contains space)
MyScript.QL             ✓ Works but inconsistent

Best Practices

Development Workflow

  1. Create Your Script
    /quill edit myScope/newfeature.ql
    - Write your code in the web editor
    - Save and close when done

  2. Load and Test
    /quill load myScope/newfeature.ql
    - Test the functionality in-game
    - Check console for any errors

  3. Iterate
    /quill reload myScope/newfeature.ql
    - Make changes in the web editor
    - Reload to see updates
    - Repeat until satisfied

  4. Deploy
    - Once tested, leave the script loaded
    - Scripts auto-load on server restart

Testing

Use Try-Catch Blocks:

OnEvent(PlayerJoin) {
    try {
        // Your event handling code
        broadcast("Welcome " + event.player.name)
    } catch error {
        console.log("Error in PlayerJoin: " + error)
    }
}

Test in Scopes First:
- Create a test scope for development
- Test scripts in isolation before deploying globally
- Use smaller region boundaries during testing

Console Monitoring:
- Watch the server console for errors
- Lexer, parser, and runtime errors are logged with line numbers
- Use console.log() for debugging output

Production Deployment

Auto-Loading:
- All scripts in the scripts/ directory auto-load on server start
- Ensure critical scripts are present before restart

Performance Considerations:
- Avoid heavy operations in high-frequency events (like PlayerMove)
- Use conditionals to filter events early
- Consider async operations for complex tasks

Security:
- Use scope security modes (whitelist/blacklist) appropriately
- Grant minimal necessary permissions to scope scripts
- Test permission boundaries before production use

Backup:
- Keep backups of working scripts before major changes
- Version control your scripts directory
- Test updates on a staging server first

Monitoring:
- Check server logs regularly for script errors
- Monitor server performance when loading many scripts
- Use /quill list to verify all critical scripts are loaded

Documentation:
- Comment your scripts clearly
- Document required permissions for scope scripts
- Note any dependencies between scripts
