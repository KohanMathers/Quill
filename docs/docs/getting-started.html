Getting Started with Quill

Getting Started with Quill

What You'll Learn

This guide will walk you through everything you need to know to start creating Quill scripts. By the end, you'll understand:

  • What scopes are and how to create them
  • How to write basic Quill scripts
  • How to respond to Minecraft events
  • How to test and debug your scripts
  • Best practices for writing maintainable code

This tutorial assumes no prior programming experience. If you're already comfortable with scripting, you might prefer the Quick Start guide instead.

Prerequisites

Before you begin, make sure you have:

  1. Server Access - You need access to a Minecraft server with Quill installed
  2. Permissions - You need either:
    - Operator status on the server, OR
    - The quill.scope.create and quill.script.load.scope permissions
  3. File Access - Ability to create and edit files in the server's plugins/Quill/scripts/ directory (or use the web editor with /quill edit)

If you're a server owner setting up Quill for the first time, see the Installation Guide.

Understanding Scopes

What is a Scope?

A scope is Quill's fundamental building block. Think of it as a container that combines:

  1. A physical region - A 3D box in your Minecraft world
  2. Scripts - Code that runs when events happen in that region
  3. Permissions - Control over which functions the scripts can use
  4. Persistent data - Variables that survive server restarts

Scopes serve two main purposes:

  • Isolation - Scripts in one scope can't interfere with scripts in another scope
  • Security - Server admins control exactly what each scope can do

For example, you might create a scope for a minigame arena where scripts can:
- Heal players and change their gamemode
- But NOT break blocks or kick players

Creating Your First Scope

Scopes are created using the /quill scope create command. The syntax is:

/quill scope create <name> <owner> <x1> <y1> <z1> <x2> <y2> <z2> <mode>

Let's break this down:

  • name - A unique identifier for your scope (e.g., my_first_scope)
  • owner - The player who owns this scope (username or UUID)
  • x1 y1 z1 - First corner of the region (coordinates)
  • x2 y2 z2 - Opposite corner of the region (coordinates)
  • mode - Security mode: whitelist or blacklist

Example: Create a 100×100×100 block scope centered at spawn:

/quill scope create tutorial_zone Steve -50 60 -50 50 120 50 whitelist

This creates a scope named tutorial_zone owned by Steve, covering coordinates from (-50, 60, -50) to (50, 120, 50), using whitelist security mode.

Tip: The coordinates don't need to be in any particular order - Quill will automatically sort them to create a proper box.

Scope Boundaries and Regions

The physical boundaries of a scope define where its scripts are active. Understanding this is crucial:

Inside the scope:
- Scripts can respond to events (player movement, block breaks, etc.)
- Functions can interact with the world
- Players can be added to the scope's player list

Outside the scope:
- Scripts don't respond to events happening outside
- Functions still work but should be used carefully

Example scenario:

// This scope covers coordinates (-50, 60, -50) to (50, 120, 50)
let my_scope = new Scope(-50, 60, -50, 50, 120, 50)

OnEvent(PlayerMove) {
    // This only triggers when players move INSIDE the scope region
    if in_region(player, my_scope) {
        sendmessage(player, "You're inside the scope!")
    }
}

Important notes:

  • Regions are inclusive - players exactly on the boundary are considered inside
  • Y-coordinates matter - make sure your region covers the full height you need
  • Scopes can overlap - players can be in multiple scopes simultaneously

Writing Your First Script

Script File Structure

Quill scripts are text files with a .ql or .quill extension. They live in your server's plugins/Quill/scripts/ directory.

For scope-specific scripts, create a folder matching your scope name:

plugins/Quill/scripts/
  ├── my_scope/
  │   └── welcome.ql
  └── another_scope/
      └── game.ql

For global scripts (not tied to a scope), place them directly in the scripts folder:

plugins/Quill/scripts/
  ├── global_events.ql
  └── my_scope/
      └── welcome.ql

Let's create our first script. Create a file at:

plugins/Quill/scripts/tutorial_zone/welcome.ql

Basic Syntax

Quill's syntax is designed to be readable and familiar. Here's a simple script:

// This is a comment - it's ignored by Quill
// Comments help explain your code

// Variables store data
let player_count = 0

// Functions contain reusable code
func greet_player(p) {
    sendmessage(p, "Welcome to the tutorial zone!")
    sendmessage(p, "Player count: {player_count}")
}

// Event handlers respond to things happening in Minecraft
OnEvent(PlayerJoin) {
    player_count = player_count + 1
    greet_player(player)
}

Key syntax rules:

  1. Comments start with // (single line) or /* */ (multi-line)
  2. Variables are declared with let (or const for constants)
  3. Statements end with a newline (semicolons are optional)
  4. Blocks use curly braces { }
  5. Strings use double quotes "like this"

Variables and Data Types

Variables store information. Declare them with let:

let name = "Steve"           // String (text)
let health = 20              // Number
let is_alive = true          // Boolean (true/false)
let nothing = null           // Null (no value)
let items = ["sword", "bow"] // List
let data = {                 // Map (key-value pairs)
    score: 100,
    level: 5
}

Variable rules:

  • Variables must be declared before use
  • Use let for variables that can change
  • Use const for values that never change
  • Variable names can contain letters, numbers, and underscores
  • Variable names are case-sensitive (player ≠ Player)

Example:

const MAX_PLAYERS = 10  // Constant - never changes
let current_players = 0 // Variable - can change

OnEvent(PlayerJoin) {
    current_players = current_players + 1

    if current_players > MAX_PLAYERS {
        kick(player, "Server full!")
    }
}

OnEvent(PlayerQuit) {
    current_players = current_players - 1
}

Event Handlers

Event handlers let your scripts react to things happening in Minecraft. They use the OnEvent() syntax:

OnEvent(EventName) {
    // Code here runs when the event happens
}

Common events:

  • PlayerJoin - When a player joins the server
  • PlayerQuit - When a player leaves the server
  • PlayerChat - When a player sends a chat message
  • PlayerMove - When a player moves
  • BlockBreak - When a block is broken
  • BlockPlace - When a block is placed

Each event provides context variables you can use:

OnEvent(PlayerJoin) {
    // 'player' is automatically available
    sendmessage(player, "Welcome, {player.name}!")
}

OnEvent(PlayerChat) {
    // 'player' and 'chat' are available
    if chat.message == "!help" {
        sendmessage(player, "Available commands: !help, !info")
    }
}

OnEvent(BlockBreak) {
    // 'player' and 'block' are available
    sendmessage(player, "You broke a {block.type} block!")
}

Your first complete script:

// welcome.ql - Greet players when they enter the scope

let visitors = 0

OnEvent(PlayerMove) {
    // Check if player just entered the region
    if in_region(move.to, my_scope) && !in_region(move.from, my_scope) {
        visitors = visitors + 1
        sendmessage(player, "Welcome! You're visitor #{visitors}")
        playsound(player, "entity.player.levelup", 1.0, 1.0)
    }
}

Testing and Debugging

Loading Scripts

Once you've created a script file, you need to load it into Quill:

For scope scripts:

/quill load tutorial_zone/welcome.ql

For global scripts:

/quill load my_script.ql

Other useful commands:

/quill reload tutorial_zone/welcome.ql  # Reload after making changes
/quill unload tutorial_zone/welcome.ql  # Stop the script
/quill list                              # See all loaded scripts

Tip: During development, keep the server console open to see error messages immediately.

Common Errors

When you're learning, you'll encounter errors. Here are the most common ones and how to fix them:

1. Lexer Error - "Unrecognised character in source"

[Quill] Lexer error in tutorial_zone/welcome.ql: Unrecognised character in source: @

Cause: You used a character Quill doesn't understand (like @ or $)

Fix: Check your syntax - you might have a typo or used the wrong symbol

2. Parser Error - "Expected IDENTIFIER but got NUMBER"

[Quill] Parser error in tutorial_zone/welcome.ql: Expected identifier but got number at line 5

Cause: Syntax error - something is in the wrong place

Fix: Check the line number mentioned. Common causes:
- Variable name starts with a number: let 1player = "Steve" (invalid)
- Missing keyword: player_count = 0 (should be let player_count = 0)
- Typo in keyword: fonction greet() (should be func greet())

3. Runtime Error - "Undefined variable"

[Quill] Runtime error in tutorial_zone/welcome.ql: Undefined variable: player_count

Cause: You tried to use a variable that doesn't exist

Fix: Make sure you declared the variable with let before using it

4. Runtime Error - "Function not found"

[Quill] Runtime error: Expected player but got string

Cause: You passed the wrong type of value to a function

Fix: Check the function documentation - you might be passing a string when it expects a player object

5. Permission Error - "Scope does not have permission"

[Quill] Scope 'tutorial_zone' does not have permission to call function 'kick'

Cause: Your scope's security settings don't allow that function

Fix: Grant the permission:

/quill scope permission grant tutorial_zone kick

Using the Log Function

The log() function is your best friend for debugging. It prints messages to the server console:

let player_count = 0

OnEvent(PlayerJoin) {
    log("PlayerJoin event triggered!")
    log("Player name: {player.name}")

    player_count = player_count + 1
    log("New player count: {player_count}")

    sendmessage(player, "Welcome!")
}

Console output:

[Quill] (tutorial_zone) PlayerJoin event triggered!
[Quill] (tutorial_zone) Player name: Steve
[Quill] (tutorial_zone) New player count: 1

Debugging tips:

  1. Use log() liberally - Log at the start of event handlers, before important operations, and after changes
  2. Log variable values - Use string interpolation: log("Score: {score}")
  3. Use descriptive messages - log("Before heal") and log("After heal") help track flow
  4. Check the scope name - The log shows which scope logged the message

Example debugging session:

// Debugging why players aren't being teleported

OnEvent(PlayerChat) {
    log("Chat event received")
    log("Message: {chat.message}")

    if chat.message == "!spawn" {
        log("Spawn command detected!")

        let spawn = location(0, 64, 0)
        log("Created location: {spawn}")

        teleport(player, spawn)
        log("Teleport completed")
    }
}

Best Practices

Naming Conventions

Good names make your code easier to understand and maintain.

Variables and functions - use snake_case:

let player_count = 0           // Good
let playerCount = 0            // Avoid (camelCase)
let PLAYER_COUNT = 0           // Avoid (SCREAMING_SNAKE_CASE)

func heal_player(p) { }        // Good
func HealPlayer(p) { }         // Avoid (PascalCase)

Constants - use UPPER_SNAKE_CASE:

const MAX_PLAYERS = 10
const SPAWN_LOCATION = location(0, 64, 0)
const GAME_DURATION = 300

Scopes - use lowercase with underscores or hyphens:

tutorial_zone      // Good
minigame-arena     // Good
TutorialZone       // Avoid

Use descriptive names:

// Bad - unclear what these mean
let x = 0
let tmp = player
let data = []

// Good - intent is clear
let player_count = 0
let target_player = player
let high_scores = []

Code Organization

1. Structure your scripts logically:

// ========== CONSTANTS ==========
const MAX_HEALTH = 20
const GAME_DURATION = 300

// ========== VARIABLES ==========
let game_active = false
let player_count = 0

// ========== FUNCTIONS ==========
func start_game() {
    game_active = true
    broadcast("Game starting!")
}

func end_game() {
    game_active = false
    broadcast("Game ended!")
}

// ========== EVENT HANDLERS ==========
OnEvent(PlayerJoin) {
    player_count = player_count + 1
}

OnEvent(PlayerQuit) {
    player_count = player_count - 1
}

2. Use comments to explain WHY, not WHAT:

// Bad - comment just repeats the code
// Increment player count
player_count = player_count + 1

// Good - comment explains the reason
// Track active players for game balancing
player_count = player_count + 1

// Good - comment explains edge case
// Don't teleport spectators - they're observing
if get_gamemode(player) != "spectator" {
    teleport(player, spawn)
}

3. Keep functions small and focused:

// Bad - function does too many things
func handle_player(p) {
    heal(p)
    set_gamemode(p, "survival")
    give(p, "diamond_sword")
    teleport(p, spawn)
    sendmessage(p, "Ready!")
}

// Good - separate functions for each responsibility
func heal_player(p) {
    heal(p)
}

func prepare_for_game(p) {
    set_gamemode(p, "survival")
    give(p, "diamond_sword")
}

func teleport_to_spawn(p) {
    teleport(p, spawn)
}

func send_ready_message(p) {
    sendmessage(p, "Ready!")
}

4. Group related scripts in folders:

plugins/Quill/scripts/
  ├── pvp_arena/
  │   ├── game_logic.ql
  │   ├── player_management.ql
  │   └── scoring.ql
  └── safe_zone/
      ├── protection.ql
      └── welcome.ql

Performance Considerations

1. Avoid expensive operations in frequent events:

// Bad - PlayerMove fires CONSTANTLY
OnEvent(PlayerMove) {
    for p in get_online_players() {  // Very expensive
        sendmessage(p, "Someone moved!")
    }
}

// Good - Use less frequent events
OnEvent(PlayerJoin) {
    broadcast("{player.name} joined!")
}

2. Filter events early:

// Bad - does unnecessary work
OnEvent(BlockBreak) {
    let block_type = block.type

    if block_type == "diamond_ore" {
        // Only care about diamond ore
        sendmessage(player, "You found diamonds!")
    }
}

// Good - exit early if not relevant
OnEvent(BlockBreak) {
    if block.type != "diamond_ore" {
        return  // Exit immediately if not diamond
    }

    sendmessage(player, "You found diamonds!")
}

3. Cache expensive lookups:

// Bad - looks up players repeatedly
OnEvent(PlayerChat) {
    if len(get_online_players()) > 10 {
        if len(get_online_players()) > 20 {
            // Called get_online_players() twice!
        }
    }
}

// Good - look up once and store
OnEvent(PlayerChat) {
    let online = get_online_players()
    let count = len(online)

    if count > 10 {
        if count > 20 {
            // Uses cached value
        }
    }
}

4. Use region checks wisely:

// Good - only process events in the region
OnEvent(PlayerMove) {
    if !in_region(player, my_scope) {
        return  // Ignore events outside our scope
    }

    // Process movement inside scope
}

5. Avoid creating objects in loops:

// Bad - creates location object 100 times
for i in range(0, 100) {
    let loc = location(i, 64, 0)
    set_block(loc, "stone")
}

// Good - more efficient if possible
let y = 64
let z = 0
for x in range(0, 100) {
    set_block(x, y, z, "stone")
}
